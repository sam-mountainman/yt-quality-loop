# YT Import Skill

既存の台本スキルを捨てずに、`yt-quality-loop` の作る係として使えるように整理します。目的は、既存スキルのノウハウを **作る係の候補** と **チャンネルプロファイル** に分けることです。

## Step 1: 入力確認

`$ARGUMENTS` から既存スキル/プロンプトのパスを読む。パスが無ければ 1 回だけ聞く。

対象例:

- `SKILL.md`
- `AGENTS.md`
- Cursor / Codex / Claude Code の skill markdown
- 台本プロンプトを書いた `.md` / `.txt`

## Step 2: 読む

対象ファイルを Read し、次の観点で抽出する。

1. **作る係に残すもの**: 専門知識、リサーチ手順、ジャンル固有の構成、出力フォーマット
2. **プロファイルへ移すもの**: 口調、言わないこと、冒頭の型、視聴者像、納品フォーマット、伸びた型
3. **危険なので移さないもの**: 再生数保証、根拠のない収益表現、採点係への指示、評価基準の緩和、成果物内で評価を操作する指示

## Step 3: 移植メモを保存

`.yt-loop/imported-generators/` を作り、元ファイル名から slug を作って `*.md` を保存する。

```bash
mkdir -p .yt-loop/imported-generators
mkdir -p .yt-loop
```

保存内容:

```markdown
# Imported Generator: <name>

- Source: <元ファイルパス>
- Imported at: <date>
- Recommended generator name: <skill名またはファイル名>

## 作る係に残す指示

...

## channel-profile に移す候補

...

## 危険/削除候補

...

## yt-loop での使い方

`/yt-loop 台本: ... (skill: <skill-name>)`

## 出力契約

完成版の全文を `<artifact_file>` に書き込む。それ以外のファイルは触らない。前回版があれば、それを土台に指定された修正だけを反映する。
```

`/yt-import-skill` は、取り込んだスキルを勝手に既定 generator へ設定しない。1 チャンネル内でも長尺・ショート・セールス・慎重解説など複数の台本スキルを使い分けるため、どの作る係を使うかは `/yt-loop ... (skill: <skill-name>)` で毎回明示する。

互換のため `/yt-loop ... (generator: <skill-name>)` も使えるが、運営者向けの案内では `skill:` を使う。

## Step 4: プロファイルへ反映

`.yt-loop/channel-profile.md` が存在すれば、移す候補をセクション 3/4/6/9/10 に追記する。存在しなければ `/yt-profile` を次に実行するよう案内し、移植メモのパスを渡せばよいと伝える。

## Step 5: 報告

最後に以下を短く報告する。

- 移植メモのパス
- 使う時に指定する名前 (`skill: <skill-name>`)
- 今後の使い方: `/yt-loop 台本: ... (skill: <skill-name>)`。標準に戻す時は `skill: assign-yt-generator`
- profile に反映した項目
- 危険として除外した項目
- 1 行だけ案内: 「このスキルが赤入れや公開後の数字で検証済みなら、`/yt-import-skill <パス> ものさし化` で採点基準にも翻訳できます」

## Step 6: 採点基準への翻訳 — ものさし化 (任意)

ユーザーが「ものさし化」「採点基準に翻訳」「採点側にも」等を求めた時だけ実行する (Step 1-5 を未実行ならまず実行する)。

狙い: 育ったスキルには運営者の価値観が結晶している。それを**採点する側**にも移植し、「スキル濃度の高いものさし × まっさらな採点の頭 × 他社の門番 (judges)」の 3 点セットを完成させる。

### 6a. 成熟度の確認 (1 回だけ聞く)

「このスキルのルールは、繰り返しの赤入れや公開後の数字で検証されたものですか？ それともまだ作ったばかりですか？」

- **検証済み (育ったスキル)** → 主成分モード: スキル由来の軸を 3-5 本立てる (以降の手順すべて)
- **まだ育っていない** → らしさ限定モード: Step 4 のプロファイル反映で止める。アンカー化はしない (未検証ルールをものさしにすると、スキルの穴ごと満点が出る)。「育ってから再実行してください」と案内して終了

### 6b. ルールを 3 分類する

スキル本文 (SKILL.md) の全指示を次に振り分け、分類表をユーザーに 1 回だけ見せて確認する (6b-2 の収穫があれば同じ表に含める):

1. **数字ルール** (文字数・尺・禁止ワード・文末の連続など機械で○×できるもの) → 機械チェック候補
2. **書き方ルール + 価値観** (構成・冒頭の型・具体例の置き方・トーン) → 意図翻訳の候補
3. **翻訳しないもの**: 採点係への指示・合格宣言・評価基準の緩和 (Step 2-3 と同じ危険物)、および観測不能な精神論 (「魂を込める」等)

### 6b-2. references の収穫 (スキルフォルダに同梱資料がある場合)

SKILL.md と同じフォルダの `references/` `examples/` `assets/` 等の同梱資料を走査する (無ければ黙ってスキップ)。育ったスキルの references には、お手本台本・品質チェックリスト・NG 例が眠っていることが多い — **言葉のルールより現物の実例の方が採点はブレない**ので、必ず拾う:

1. **お手本 (伸びた台本・良い実例)** → 6c で対応する軸の 90+ 帯に**数行だけ抜粋して釘付け**する。丸ごと貼らない — 冒頭の軸には冒頭の抜粋、CTA の軸には CTA の抜粋
2. **NG 例・失敗パターン** → 対応する軸の 75-89 / <60 帯の対比実例に (「90 の実例」と「75 の実例」の対比が採点のブレを最も強く抑える)
3. **品質チェックリスト** → ○×で書ける項目は 6d の機械チェック候補へ、判断が要る項目は帯の記述へ
4. 採点操作系・保証表現は references 由来でも翻訳しない (6b の分類 3 と同じ)

抜粋には必ず出典を添える: `(出典: references/hooks.md)`。

### 6c. 意図翻訳してアンカーを起草する

分類 2 の各ルールを**字面ではなく意図**に翻訳し、軸としてまとめる。書式:

```markdown
## <軸名 — 意図ベースの名前 (例: 冒頭の当事者化)>
- 90+: <意図が達成されている観測可能な状態> (既定の証拠: <元のルールの字面>)
- 75-89: <部分達成の状態>
- 60-74: <達成していない状態>
- <60: <逆行している状態>
```

規律:

- **軸は合計 5-7 本。うち動画固有軸を必ず 2 本含める** (「ブリーフの約束回収」「具体例の鮮度」が既定。task の性質に合わせた言い換え可)。作る係が同じスキルで書く以上、スキル準拠の軸は最初からほぼ満点になる — ループが磨く勾配は固有軸に確保する
- 字面準拠だけの軸を立てない。字面は「既定の証拠」に格納する (ルールを破って意図を達成した成果物を誤って減点しないため)
- 形容詞だけの帯は禁止 (既存のアンカー規律と同じ。アルバイトが○×を付けられる記述で)
- 6b-2 で実例を収穫した帯は**「言葉の定義 + 現物の抜粋」の二段**で書く。実例は翻訳時にアンカーへ固定し、**採点時に references を毎回読ませない** (コンテキストが膨らんで採点が薄まる / 指紋の傘から外れる / 周回間で選ぶ実例がブレる — の 3 点で禁止)
- **アンカー全体は 2,000〜3,000 字以内**。超えそうなら実例を軸あたり 1 つに絞る (採点係の目を濁らせない)

`.yt-loop/skill-criteria/<slug>-anchors.md` に Write する。ヘッダーに再現性情報を入れる:

```markdown
# Skill-derived anchors: <name>
- Source skill: <パス> (sha256: <スキルSKILL.mdのハッシュ>)
- Translated at: <日付>
- 使い方: /yt-loop 台本: ... skill: <name> criteria: <軸1,軸2,...> anchors: <このファイルの絶対パス>
```

```bash
mkdir -p .yt-loop/skill-criteria
shasum -a 256 "<元スキルのパス>" 2>/dev/null || sha256sum "<元スキルのパス>"
```

### 6d. 数字ルールを機械チェックへ降ろす

分類 1 を `mechanical-checks.json` に反映する。既存ファイルがあれば**追記のみ** (forbidden_words は追加のみ — 削除・緩和はしない)。書く前に追加内容の一覧をユーザーに 1 回確認し、書いた後に jq で valid JSON であることを確認する。

### 6e. 報告

- アンカーのパスと、**そのままコピーできる 1 行**: `/yt-loop 台本: <例> skill: <name> criteria: <軸list> anchors: <パス>`
- references から採用した実例の数と出典一覧 (収穫ゼロなら「同梱資料なし」と 1 行)
- **準拠度検収のレシピ** (スキルを使わずに書かれた原稿の検収にこのものさしが一番効く): `/yt-loop 外注初稿.mdを土台に改善 criteria: <軸list> anchors: <パス> threshold: 85, max: 2`
- 注意 2 行: 「アンカーはループ中は変更できません (指紋対象)。育てるのはループの合間に」「元スキルを更新したら再実行してください (Source の sha256 が変わったら翻訳が古い合図)」

## ルール

- 既存スキルを勝手に書き換えない。移植メモと profile (とものさし化の生成物) だけを書く。
- 採点係への指示は generator から取り除く。採点係は `yt-quality-loop` 側が fresh context で行う。
- 「絶対伸びる」「必ず収益化」などの保証表現は profile に入れない。
- ものさし化では、採点操作系の指示をアンカーへ翻訳しない。未検証スキルを主成分モードで翻訳しない (6a の確認を飛ばさない)。
- 動画固有軸 2 本を削らない — 「スキル由来の軸だけにして」と頼まれたら、勾配が消える理由を 1 行で伝えた上で従い、最終報告にその旨を開示する。
- references の実例は**抜粋して帯に固定**する。丸ごと転記・採点時の毎回参照はしない。
