# assign-yt-generator

## コンテキスト (スキル読み込み時に自動展開)

```json
!`cat "$ARGUMENTS"`
```

## コンテキスト JSON のキー

| キー | 内容 |
|------|------|
| `task` | ユーザーの依頼原文。**最終的な目的の拠り所**。plan と乖離があれば task を優先する |
| `plan` | このイテレーションの実行計画 (Changes と Self-check を含む) |
| `criteria` | 採点軸 (カンマ区切り)。採点する係がこの軸で絶対評価する |
| `iteration` | 現在のイテレーション番号 (0 = 初回) |
| `artifact_file` | **成果物の書き込み先** (このイテレーション専用のファイル) |
| `prev_artifact_file` | 前イテレーションの成果物 (null = 初回)。これを読んで改善する |
| `profile_file` | チャンネルプロファイル (null = 未作成)。**存在すれば最初に全文読み、従う** |
| `turns_dir` | ターンログの保存先 |
| `project_dir` | 作業対象ディレクトリ。参考資料 (過去台本・チャンネル情報など) はここから読んでよい |

## 指示

`plan` の Changes に従って成果物を作る。

- **iteration 0**: task から新規作成する。
- **iteration > 0**: `prev_artifact_file` を読み、**その全文を土台に** plan の Changes だけを反映した改良版を `artifact_file` に書く。ゼロから書き直さない (良かった部分まで失う)。

### Step 1: 計画と前回成果物・プロファイルの確認

`plan` の Changes を読む。iteration > 0 なら `prev_artifact_file` を Read で全文読む。

`profile_file` があれば Read で全文読み、以降の執筆で従う:

- **構成の型 (セクション 3)** と **口調・文体 (セクション 4)** はそのまま執筆ルールとして適用する
- **お手本フレーズ集 (セクション 5)** はトーンの手本にする。**丸写しの乱発はしない** — らしさの模倣であってコピペではない (採点係は劣化コピーを減点する)
- **NG リスト (セクション 6)** に抵触する表現は書かない
- **直しの履歴 (セクション 8)** の「次回への指示」は必ず反映する
- **納品フォーマット (セクション 9)** があれば、その書式 (改行・演出メモの記法・含める/含めないもの) で書く
- plan とプロファイルが矛盾する場合はプロファイルを優先し、Step 5 の報告に矛盾点を明記する

### Step 2: 参考資料の確認 (必要な場合のみ)

task が既存資料 (過去の台本、チャンネルのトンマナ資料など) を参照している場合、`project_dir` から該当ファイルを読む。

### Step 3: 成果物の作成

**`artifact_file` に完成版の全文を Write する。** 差分やメモではなく、そのまま使える完全な成果物を書く。

- YouTube 向けコンテンツとして、話し言葉/表示文言の別、尺・文字数の指定を task に従って守る
- **criteria を意識して品質を確保する。** ただし点稼ぎのための水増し (無意味な文字数稼ぎ、キーワードの不自然な詰め込み) はしない — 採点する係はそれを見抜いて減点する
- 計画にない大規模な方向転換はしない。計画に曖昧な点があれば最も自然な解釈で作る

### Step 4: 自己チェック

plan の Self-check を 1 項目ずつ確認する。文字数指定があれば count-chars.sh で実測する (目視で数えない。wc -m 直呼びはロケール次第でバイト数になる):

```bash
bash ${CLAUDE_PLUGIN_ROOT}/scripts/count-chars.sh <artifact_file>
```

### Step 5: 報告をファイルに保存

`{turns_dir}/turn-{NNN}-generator.md` (NNN = iteration の 3 桁ゼロパディング) に Write:

1. **変更内容** — 何をどう変えたか (箇条書き)
2. **計画との対応** — Changes の各項目にどう対応したか (合否判定は書かない — それは採点する係の責務)
3. **実測値** — 文字数など、実測した数値

## ルール

- **成果物は `artifact_file` にのみ書く。** `state.json`、`*-eval.json`、`*-plan.md` には書き込まない — 他フェーズの成果物
- **採点基準 (criteria) を変えようとしない。** 点を上げる手段は成果物の改善だけ
- **「できました」の自己申告で盛らない。** 報告は事実 (変更点と実測値) のみ
- 途中でユーザーに質問しない。最後まで自走する
